### Fine-Tuning Decision Framework (Notebook 9)

- Go/no-go logic: fine-tune only when a well-constructed prompt still misses the mark
  - “Well-constructed” means further iteration degrades quality, not just plateaus
  - Current situation: model is decent but not great, dataset is strong, so fine-tuning is warranted
- Prompt engineering should be exhausted first
  - Detailed system prompts (2-4x longer, structured markdown, explicit formatting rules) can close ~80% of the gap
  - DSPy library automates prompt optimization via evolutionary search: vary, test, keep best, repeat
  - Model choice is also a variable: swap models before committing to fine-tuning
- Results saved as prompt_engineering_baseline.json for downstream eval

### LoRA and Quantization (Notebook 2 Walkthrough)

- Training setup: 2,000 training samples, 100 eval samples, Hugging Face for adapter storage
- LoRA config defaults:
  - r = 16 (balance), alpha = 32 (always 2x r), dropout = 5% (best practice, don’t adjust)
  - Higher r = more adaptation capacity; reduce r if overfitting
- Quantization: 16-bit floats compressed to 4-bit, preserving 98-99% of model accuracy
  - Saves 75% of model size; quantized versions typically released weeks after base model
- Tokenization: each model uses its own token vocabulary; tokenizer library translates the universal messages array into model-specific tokens
  - Fine-tuning changes how tokens are internally represented (vectors/weights), not the vocabulary itself
  - LoRA adds a small delta to select weights rather than modifying the base model directly

### Debugging Overfit Models

- Print training loss and eval loss every ~100 steps
  - Both declining together: healthy
  - Eval loss rising while training loss falls: classic overfitting
  - Eval loss flat/high regardless: training and eval criteria are misaligned
- Knobs to adjust if overfitting:
  - Reduce r (and alpha follows automatically)
  - Lower learning rate or reduce epochs
  - Learning rate scheduler: can help, but starting lower is often simpler
  - Dropout: leave at 5%, not effective for this problem

### PEFT and Adapter Workflow

- PEFT library snaps base model and LoRA adapter together like “legos,” no permanent merge needed
- Adapter must be pushed to Hugging Face before session ends, or it is lost
- GPU requirement: T4 often insufficient; A100 or local IDE preferred
  - RunPod is a paid workaround if Colab fails
- Inference notebook: toggles adapter on/off to compare base vs. fine-tuned output side by side

### When Fine-Tuning Actually Makes Sense

- Fine-tuning is rare in practice; prompting gets most of the way there most of the time
- Truly shines on domains the model has never seen
- At a startup moving fast: not worth the investment unless the agent survives more than a month
- This notebook is more tutorial than real-world prescription
