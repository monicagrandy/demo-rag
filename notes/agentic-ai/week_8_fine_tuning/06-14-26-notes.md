### When to Fine-Tune vs. Alternatives

- Decision hierarchy: prompt engineering first, then RAG, then fine-tuning as last resort
- Prompt engineering
  - Gets 80% of the way there in minutes, very low cost
  - Wins when the model already knows the domain and needs behavioral guidance
- RAG
  - Wins when you need up-to-date facts, source citations, or document-grounded answers
  - Requires vector DB and embeddings setup (days of effort, but reversible)
- Fine-tuning
  - Wins when style, terminology, and format must be baked into model weights
  - Only escalate when there is measured evidence the current approach fails
  - Fine-tuning teaches new habits, not new knowledge (model likely already has domain knowledge from pre-training)

### Full Fine-Tuning vs. Parameter-Efficient Fine-Tuning (PEFT)

- Full fine-tuning: updates all parameters of the base model
  - Slow, RAM-intensive, large output file, high risk of catastrophic forgetting
  - Irreversible in practice: changing all weights unpredictably alters existing knowledge
  - Best suited to frontier model creators (OpenAI, Anthropic, Alibaba), not domain adaptation
- PEFT (LoRA / QLoRA): adds a small trainable adapter layer on top of the frozen base model
  - Analogy: painting the wall (full) vs. putting a poster on it (PEFT)
  - Trains ~1% of parameters; much faster, less RAM, smaller output file
  - Adapters are modular: stack, swap, or remove without touching the base model
  - Quality retained at 95-99% vs. full fine-tuning

### LoRA and QLoRA Mechanics

- LoRA (Low-Rank Adaptation): parallel low-rank A/B matrix added to frozen layers
  - Runs in parallel, so no added inference latency (unlike sequential adapter layers)
- QLoRA: LoRA + 4-bit quantization of the base model
  - Reduces model size by 75% (4-bit vs. 16-bit), with ~1-2% quality loss
  - Weights renormalized over remaining distribution (not just truncated)
  - Double quantization saves an additional ~1.1 GB on a 1.5B parameter model
  - Enables training on Google Colab free tier (Tesla T4 GPU)
- Key LoRA hyperparameters
  - r (rank): learning capacity; higher = more adaptation power, higher overfit risk
  - alpha: scaling constant, typically set to 2x rank
  - dropout: ~5% neurons dropped per step to force redundant connections
  - target_modules: “all linear” targets every layer (standard default)
- Quantization config must match exactly between training and inference; mismatch breaks the adapter entirely

### Data Quality and Dataset Preparation

- Two datasets used in the healthcare LLM experiment:
  - ChatDoctor: 112,000 examples, heavily contaminated
    - 63% contain persona phrases (“Hi, welcome to ChatDoctor!”) and boilerplate sign-offs (“Wishing you good health”)
    - Almost no safety disclaimers
  - WikiDoc: 2,000 examples, clean and well-structured
    - Zero persona contamination, zero boilerplate, 99.4% include safety disclaimers
- Result: WikiDoc (2K clean) outperforms ChatDoctor (112K noisy) on all evaluation metrics
- Data sourcing approaches and trade-offs:
  - Synthetic (LLM-generated): fast and flexible, but hallucination and formatting errors require manual review anyway
  - Web/internal scraping: free but very time-consuming
  - Expert annotation (e.g., Amazon Mechanical Turk): high quality, expensive ($2K-$10K per dataset); annotators are tested on a hidden labeled subset before labeling the full set
  - Production logs: highest real-world quality, but requires PII scrubbing and formatting work
- Audit before training: check for persona phrases, boilerplate, and safety disclaimer rates before committing to a dataset
- Temperature note: 0.9 used for synthetic generation increases diversity but raises hallucination risk

### Model Choice and Training Setup

- Model: Qwen 1.5B parameter instruct (open source, commercial-safe, fits on Colab T4)
  - Already pre-trained on medical textbooks, PubMed, Wikipedia; fine-tuning teaches style, not new facts
- Chat template: each model (Qwen, DeepSeek, Kimi) uses different special tokens; must apply the correct tokenizer via tokenizer.apply_chat_template
  - System prompt must be identical between training examples and inference; mismatch degrades output quality
- Training harness hyperparameters:
  - Epochs: up to 3 passes over the training data
  - Learning rate: lower for WikiDoc (precise style changes), higher for ChatDoctor (to demonstrate overfitting)
  - LR scheduler: decays over time so early steps take large gains, later steps converge precisely
  - Warm-up steps: ramp up before full learning rate kicks in
  - Save steps: checkpoint every 100 steps, enabling rollback if final model overfits
- Versioning: pin library versions carefully; ML library dependency conflicts are a leading source of errors
- Adapter saved to Hugging Face Hub (~40 MB) separately from the base model (~3 GB at FP16, ~750 MB quantized)

### Evaluation with LangSmith

- Evaluation is non-negotiable: without it, model quality is assessed on vibes
- Benchmark approach: 10 hand-curated medical prompts run against base model and fine-tuned model before and after training
- Three evaluators defined: helpfulness, medical accuracy, safety (scored 0-1, normalized from 1-5)
- LLM-as-a-judge pattern (GPT-4o mini used here)
  - Most common pattern, easy to set up
  - Significant weaknesses: non-deterministic, same prompt can score 3/5 then 5/5 then 4/5
  - Risk of mutual hallucination: bad answer scored highly by a hallucinating judge
  - Better alternatives: deterministic string checks, semantic vector distance vs. ideal answer, or predefined evaluation function libraries
  - Rule: never use the same model as both subject and judge
- Results:
  - ChatDoctor fine-tune: all three metrics regressed vs. base model
  - WikiDoc fine-tune: accuracy and safety improved; helpfulness dropped slightly because the metric rewarded verbosity and the model learned to be brief
- Key insight on helpfulness drop: the evaluation metric was flawed, not the model; brevity in WikiDoc training data became brevity in responses, which is functionally desirable for a medical bot

### Key Takeaways

- What fine-tuning is: taking a pre-trained LLM and training it further on domain-specific data so it learns specialized behavior (accuracy, style, format) without training from scratch
- When fine-tuning is needed vs. not:
  - Benchmark prompt-only vs. RAG vs. fine-tuned on a labeled test set; only escalate with measured evidence of failure
  - “Irreversible” means the fine-tuned model can’t easily unlearn bad data; the original base model is always available, but fixing a bad fine-tune means retraining
- Consistent output format: prompt engineering is usually sufficient; fine-tune only when formatting is unreliable at scale across many prompt variations
- Full vs. PEFT framing: full fine-tuning creates a specialized derivative of the base model; PEFT updates a small subset of parameters and leaves the base model intact
- LoRA rank ®: number of “directions” the model can learn; approximates weight changes via two small matrices of rank r rather than updating the full weight matrix
- Deployment: adapter toggle pattern allows multiple adapters on one base model for A/B testing or stacking behaviors; quantization config must match at inference time
