### Multi-Agent System Architecture Overview

- Travel planning agent with orchestrator-worker-synthesizer format
  - Orchestrator: Query understanding layer with LLM routing requests
  - Search agent: Uses flight/hotel search tools via SERP API
  - Itinerary agent: Uses Tavily search API + deep research sub-agent
  - Synthesizer: Collates parallel agent responses into unified output
- Workflow as step-by-step computation graph
  - Each yellow circle = computation unit (LLM call, agent call, or Python function)
  - Message passing between nodes via shared state
  - Can replace any computation step with different logic types

### Agent vs Sub-Agent Design Patterns

- Agent: Complete handoff with state updates and knowledge transfer
  - Full control over project step with own tools/prompts
  - Updates graph state before moving to next node
- Sub-agent: Expert consultation without handoff
  - Deep research tool used within itinerary agent as sub-agent
  - Maintains own tools/prompts but no state transfer
  - Parent agent continues processing with sub-agent insights
- Tool reusability across multiple agents for common functions (authorization, web search)

### LangGraph Implementation Details

- Graph state definition with message passing
  - Messages array: Human, system, AI, tool message trajectory
  - User query, agent tasks, agent results attributes
  - Schema validation via Pydantic classes for tool inputs
- Two implementation approaches:
  1. Agent wrapper: Tool call + execution in single step
  2. Raw LLM: Separate tool calling and execution nodes
- Conditional edges for workflow navigation
  - Dispatch function determines next node(s) based on computation
  - Can route to single node (dictionary) or multiple nodes (list)

### Persistence and State Management

- Checkpointing capabilities for workflow interruption/resumption
  - In-memory saver: Session-specific storage
  - Database storage: Cross-session memory retention
- Snapshot functionality
  - get_state(): Latest state message with all attributes
  - get_state_history(): Complete trajectory of state evolution
- Use cases:
  - Human-in-the-loop workflows requiring user input mid-process
  - Debugging and evaluation of workflow steps
  - Context preservation for follow-up sessions
