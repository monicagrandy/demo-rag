### Multi-Agent Systems Overview

- Multi-agent systems coordinate specialized components to tackle complex workflows
- Use only when single agent systems don’t suffice - always try single agent first
- Key benefits:
  - Specialized knowledge without overwhelming model context window
  - Distributed development with clear boundaries
  - Parallelization capabilities
  - Better scalability and resilience

### When to Build Multi-Agent Systems

- Too many tools for single agent to manage effectively
- Specialized knowledge requirements with extensive context
- Need for sequential constraints or specific workflow enforcement
- Context window management becomes critical issue

### Multi-Agent Architecture Patterns

- **Network (Peer-to-Peer)**: Every agent communicates with every other agent
  - Example: Incident response where AlertAgent → AnalysisAgent → RemediationAgent can all intercommunicate
  - Flexible and resilient but hard to debug and non-deterministic
- **Supervisor (Hub-and-Spoke)**: One orchestrator routes to specialized subagents
  - Subagents report back to supervisor, never to each other
  - Clean, observable, easy to reason about
  - Supervisor becomes bottleneck and single point of failure
- **Hierarchical**: Multi-level tree with supervisor of supervisors
  - Top-level orchestrator delegates to mid-level supervisors managing subagents
  - Scales well with clean domain separation
  - Latency compounds across levels
- **Pipeline (Sequential)**: Each agent has one job, passes output as input to next
  - Linear flow, no branching - easiest to reason about
  - No parallelism, one slow stage blocks everything
- **Handoff/Swarm**: Agents pass control dynamically based on context
  - Very flexible, handles complex branching
  - Hardest to debug and predict

### Quick Decision Matrix

- **Network**: Use when agents are truly peer tasks needing resilience; avoid when need predictability
- **Supervisor**: Use for clear hub-and-spoke scenarios; avoid when too many subagents overload supervisor
- **Hierarchical**: Use for large org-like complexity; avoid when latency is critical
- **Pipeline**: Use for sequential steps with clear data flow; avoid when need parallelism
- **Handoff/Swarm**: Use for dynamic flows; avoid when need determinism

### LangGraph in Plain Terms

- **State = The Shared Whiteboard**: Python dict that every node can read/write - the memory of entire workflow
- **Nodes = Functions**: Python function taking current state, doing something, returning state update
- **Edges = Routing Logic**: Decide which node runs next (fixed or conditional)
- **Reducers**: Answer when node updates state - replace old value or merge with it?
  - Without reducer: last write wins, lose previous messages
  - With reducer: both get appended, keep full history
  - Annotated[list, add_messages] means append, don’t replace

### Travel Planning System Architecture

- Built orchestrator-style workflow with specialized agents:
  - Search Agent: handles flight and hotel search with specific tools
  - Itinerary Agent: creates detailed travel plans using deep research
  - Synthesizer: combines outputs when both agents triggered
- State flows through graph like water through pipes
- Each node transforms state, edges direct where it goes next

### Flight & Hotel Search Agent: Block by Block

- **SerpAPI Integration**: Wraps Google’s search products as structured APIs
- **Pydantic Schema Validation**: FlightSearchInput/HotelSearchInput prevent agent hallucination
  - Explicit input contracts with required fields
  - Field descriptions help LLM populate correctly
- **Tool Implementation**:
  - @tool decorator with args_schema for validation
  - Returns structured JSON with top 10 flight options or top 5 hotels
- **Clarification Gate**: Prompt prevents partial execution with assumed data
  - “If ANY required field missing, ask clarification and STOP”
  - Airport code normalization built in
- **Agent Creation**: Uses create_agent (not create_deep_agent) for bounded, deterministic search

### Key Technical Distinctions

- **Model Performance**: Primarily training-time story (data composition, RLHF, architecture) rather than inference-time context
- **Agent vs Tool Calling**: Two approaches in LangGraph
  - Wrapper method: Tool calling and execution happen in single step
  - Raw method: Separate tool calling and tool execution nodes for granular control
- **Sub-agents**: Agents used as tools within other agents
  - Wrap agent in @tool decorator to use as sub-agent
  - Provides specialized expertise without complete handoff

### Debugging and Observability

- Test individual components before testing entire workflow
- LangSmith provides trajectory visualization for complex systems
- Logging all outputs helps track which components triggered when
- System instructions quality crucial for preventing infinite loops

### Cost and Performance Considerations

- Microsoft cybersecurity case: 30 minutes → 30 seconds, $15 → $1 per incident
- Cost optimization through inference optimization, hardware optimization, prompt optimization
- Specialized smaller models (SLMs) can reduce costs while maintaining accuracy
- Latency typically higher than traditional systems but improving with better deployment strategies
