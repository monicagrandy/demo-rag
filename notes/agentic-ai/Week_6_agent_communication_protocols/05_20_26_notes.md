### MCP Protocol Deep Dive

- Solves integration complexity vs hard-coding or REST APIs
  - Example: 100+ property listing websites = 100 integrations to maintain
  - MCP provides standardized protocol for agent-tool communication
- MCP server structure:
  - Tool names in natural language
  - Input/output schemas clearly defined
  - Authentication and error handling built-in
- Agent benefits:
  - No integration code required
  - Runtime tool discovery through natural language
  - LLMs construct requests from schema definitions

### Google ADK Framework Components

- Session database (SQLite by default, pluggable to Redis/others):
  - Sessions: conversation instances
  - Events: all conversation actions (messages, tool calls, responses)
  - User states: data persisting across sessions (prefix with “user”)
  - App states: application-level data storage
- Built-in orchestrations:
  - Sequential agents (run in order)
  - Parallel agents (concurrent execution)
  - Loop agents (iterative processing)
- Callbacks: functions triggered before/after LLM calls or tool calls
- Agent-as-tool capability: agents can call other agents

### Agent-to-Agent (A2A) Protocol

- HTTP-based agent discovery and communication
- Agent cards: JSON schemas defining agent capabilities
  - Skills, message types, endpoints
  - Discoverable through central repository
- Context IDs maintain conversation state across multi-turn interactions
- Future vision: agent marketplace for specialized task delegation
- Google pushing standard, others may follow

### Technical Q&A Highlights

- Google ADK vs LangGraph: both viable, ADK provides more out-of-box features
- Open source alternatives (OpenCode.AI) offer vendor independence
- MCP responsibility lies with API owners (e.g., Zillow creates MCP server)
- Exercise clarification: use repository exercises, not drive folder modules
