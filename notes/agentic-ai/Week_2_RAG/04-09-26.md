### Embeddings Fundamentals

- Input text → vector output where semantically similar inputs produce similar vectors
- Dataset: 20 support tickets with title, description, resolution, category, priority
- Query process:
  1. Embed user query using same model as documents
  2. Calculate cosine similarity against document embeddings
  3. Return top K most similar results
- Example queries demonstrated:
  - “Database running very slow” → matched performance tickets correctly
  - “Users can’t log in after changing password” → matched authentication issues

### Similarity Scoring & Thresholds

- Cosine similarity scores vary by embedding model:
  - Google models: range 0.001 to 0.95+ (wide distribution)
  - Open source models: typically >0.6 (narrow range)
  - OpenAI models: demonstrated 0.4-0.8 range
- Threshold selection process:
  1. Test 100-500 queries to establish baseline
  2. Analyze documents just below threshold for relevance
  3. Balance precision vs recall based on use case
- RAG systems prioritize recall over precision
  - Better to retrieve irrelevant docs (LLM can ignore) than miss relevant ones

### Batch vs Single Embeddings

- Batch processing significantly faster for initial DB creation
  - Single: 2.11 seconds for 20 documents (~100ms each)
  - Batch: 0.93 seconds for same 20 documents (2.3x speedup)
- OpenAI limitations:
  - Max 2,048 inputs per batch call
  - Max 300,000 tokens total per batch
- Use cases:
  - Batch: Initial vector DB creation
  - Single: Real-time user queries

### Chunking Strategies

- Created single string per ticket: ID + title + category + priority + description + resolution
- Chunk size experiments:
  - 200 characters → 60 chunks (some warnings for oversized chunks)
  - 500 characters → 39 chunks (no warnings)
- Metadata preserved for filtering:
  - Ticket ID, category, priority
  - Enables category-specific searches

### Vector Database Operations (ChromaDB)

- Default distance metric: L2 (Euclidean) - expensive due to squaring/square root
- Recommended: Cosine similarity with HNSW algorithm
  - HNSW: Google’s approximation algorithm for fast similarity at scale
  - Handles millions/billions of documents in <5ms with minimal quality loss
- Metadata filtering capabilities:
  - AND/OR filters on category, priority, etc.
  - Example: category=“authentication” AND priority=“high”
- Persistence options:
  - Save to disk for sharing/backup
  - Load from storage to avoid re-embedding costs

### Index Types Comparison

- **Vector Index**: Best balance of build time and query speed
- **Tree Index**:
  - Hierarchical structure with configurable num_children (default 5)
  - Child branch factor determines documents selected at each level
  - Example: 20 docs → 4 root nodes with num_children=5
- **Keyword Index**:
  - Extracts keywords, matches exact terms
  - Slower build time (calls LLM for extraction)
  - Fast query time for exact matches
- **Summary Index**: Fast build, very slow queries (processes all docs each time)

### Production Considerations

- **Hybrid Retrieval**: Combine vector + keyword retrievers for best results
- **Metadata Filtering**: Essential for multi-tenant systems
- **Persistence Strategy**:
  - Daily re-indexing during low-traffic periods
  - Snapshot previous versions for rollback capability
  - Cold storage for scaling and disaster recovery
- **Client-side Processing**:
  - Viable for <5MB indices (saves server costs)
  - CORS policy prevents direct API calls from client
  - Good for privacy positioning but limits real-time updates
