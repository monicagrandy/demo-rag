### Week 2 Review: RAG Fundamentals

- **Fine-tuning vs RAG comparison**
  - Fine-tuning: modifies the model itself by updating weights using training data
    - Useful for style, behavior, or specialized tasks
    - “Teach the brain”
  - RAG: doesn’t change the model, fetches external data at runtime and injects into prompts
    - “Give open-book access”
    - More accurate for pure prompting and cheaper than fine-tuning
  - Fine-tuning preferred when behavioral adaptation needed (e.g., adding empathy to medical responses)
- **Temperature control for randomness**
  - Controls randomness in token selection
  - Low temperature (0-0.3): deterministic, sticks to high-probability tokens → more factual, less hallucination
  - High temperature (0.7+): more diverse/creative → higher risk of inventing details
  - **Keep temperature at 0.2 for RAG systems (must be less than 1)**
  - Higher temperature increases hallucination chance
- **Offline vs Online RAG processing**
  - Offline RAG: data preprocessed, chunked, embedded, and indexed ahead of time
    - Faster, cheaper
    - Works well for stable data
  - Online RAG: data fetched or updated at query time (APIs, web, DBs)
    - More up-to-date, dynamic
    - Slower, higher latency

### The Hallucination Problem

- **Primary reason for using RAG instead of pure LLM**
  - LLMs prone to inventing facts when lacking specific knowledge in training data
  - RAG provides “ground truth” context from organizational data
- **Technical reasons why large context windows increase hallucination**
  1. “Lost in the Middle” phenomenon: LLMs better at retrieving info from beginning/end of prompts
  2. Attention dilution: Multi-Head Attention strength gets diluted by noise of surrounding irrelevant tokens
  3. Noise as signal: Imprecise retrieval pulls tangentially related but factually different chunks
  4. Contextual drift: Larger contexts create opportunities for model to drift from original intent
- **Confidence vs correctness detection**
  - High-confidence wrong signals: fluent, detailed, specific answers with no hedging
  - Low-confidence signals: hedging language, vague answers, multiple alternatives
  - Better approaches: self-consistency checks, groundedness scoring in RAG, verifier models/guardrails

### LangChain Framework

- **Core ecosystem for working with LLMs**
  - 133,000 GitHub stars - de facto standard
  - Four core “Lego blocks”: Models (LLMs), Prompts, Vector Stores, Chains (LCEL)
  - LCEL (LangChain Expression Language) uses pipe operator (|) to connect blocks
  - Makes experimentation fast through abstraction
- **Key advantages**
  - Example: Walmart breaking ties with OpenAI only requires changing model component
- **Context window management**
  - 128k input context window essentially eliminates input limits
  - 16k output window with controllable limits
  - Chat history capping standard practice: keep last N messages or cap by token count
  - Conversational history supported out-of-the-box with rewriting standalone questions

### RAG Pipeline Implementation

- **Standard RAG workflow**
  - User query → embedding → retriever → documents → LLM
  - Retriever finds docs, prompt formats them, model generates answer, parser cleans output
  - chain.invoke(question) leverages entire LangChain pipeline
- **Key components**
  - GPT-4o-mini most popular model for RAG due to cost/performance balance
  - MMR (Maximal Marginal Relevance) for diversity in top results
  - Cosine similarity score for semantic matching
  - Hybrid search combines keyword and vector matching for best results
- **Tool creation and management**
  - Two methods: @tool decorator or direct tool() function
  - Tools are simple Python functions with clear names and descriptions
  - LLM reads function name and documentation to understand tool purpose
  - Avoid tool overlap and keep number manageable

### Evaluation Metrics

- **Layer 1: Retrieval Evaluation**
  - **Precision @ K**: “Out of K documents retrieved, how many were actually useful?”
    - Low precision = too much noise, LLM gets distracted
    - Fix: improve embedding model or use metadata filtering
  - **Recall @ K**: “Of all useful documents that exist, how many did we find?”
    - Low recall = missing info, answer exists but search didn’t find it
    - Fix: increase K value or improve chunking
  - **F1 score**: harmonic mean of precision and recall
  - **Most of the time recall is more important than precision**
- **Layer 2: Generation Evaluation**
  - Target: > 0.85
  - Failure = hallucination
- **Critical thresholds for production**
  - Block deployment if: Groundedness < 0.60, Precision@3 < 0.50, Recall@3 < 0.40
  - Don’t go to production with recall < 0.4

### Failure Patterns and Solutions

- **Common failure patterns**
  - Solution: increase K value or improve chunking
- **Human vs LLM evaluation trade-offs**
  - SMEs (Subject Matter Experts) needed to determine relevance of returned information
  - Humans: high accuracy, low bias (if structured), but expensive and slow
  - LLM evaluators: cheap, scalable, but probabilistic and biased
  - Best practice: hybrid loop using LLMs for bulk evaluation, humans for edge/critical cases

### Agentic RAG

- **When to use traditional vs agentic RAG**
  - Traditional RAG: good for fast prototyping, single data source, simple queries
  - Agentic RAG: multiple sources of retrieval, complex queries requiring multi-step reasoning
  - Traditional RAG for V0 when collecting customer information/developing ICP
- **ReAct workflow pattern**
  1. **Reason**: Think which tool call to make
  2. **Act**: Decide input and make tool call
  3. **Observe**: Look at output and determine if enough information available
  4. **Repeat**: Continue until sufficient information gathered
- **Tool design principles**
  - Write perfect function names and descriptions - LLM reads these to understand purpose
  - Avoid tool overlap and keep number manageable
  - Tool descriptions are actually code - be specific about when to use each tool
  - Example tools: SearchSimilarTickets, GetTicketByID, SearchByCategory, GetTicketStats
- **Scope limitation strategies**
  - Strict system prompt: “Only answer from retrieved ticket data”
  - Grounding enforcement: reject answers without citations
  - Retriever filtering: restrict to specific databases only
  - Tool constraints: allow only specific tools
  - Guardrails: block out-of-domain queries

### Production Considerations

- **Business case evaluation**
  - RAG most powerful when replacing human responsibilities (e.g., customer care bot)
  - Search bots often don’t outperform traditional search bars
  - Example: Walmart’s Sparky performs worse than standard search bar on business metrics
  - AI implementations often driven by investor expectations rather than business value
- **Cost and infrastructure management**
  - Token utilization: generation typically bigger cost than retrieval
  - Context window management: 128k tokens allows months of conversation
  - Update mechanisms: Kafka event streams for real-time or periodic reindexing (every 24 hours standard)
  - Multi-user systems: separate chat threads, context windows not shared between users
- **Integration resources**
  - LangChain GitHub repository and documentation available at https://chain.in
  - Artificial analysis hallucination benchmark for model comparison
  - OpenAI tokenizer and models documentation for understanding token usage
